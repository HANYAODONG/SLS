# 大模型接入实现方案

日期：2026-06-14

## 1. 背景与目标

当前项目已经完成 SLSforASVspoof-2021-DF 主模型的复现工作，并在 ASVspoof 2021 DF 子集上跑通了完整流程：

```text
音频数据 -> XLS-R 300M -> SLS 分类器 -> score 文件 -> 官方 key 计算 EER
```

当前已经具备的能力包括：

- 使用 `MMpaper_model.pth` 进行 DF 音频反欺骗推理。
- 支持 10 条、5000 条、20000 条 DF 子集测试。
- 20000 条子集 EER 为 `1.99%`，与仓库报告的完整 DF EER `1.92%` 接近。
- 已经生成并维护了对应的 protocol、key 子集和运行脚本。

下一步计划是在项目中接入大模型能力，用于扩展当前系统的可解释性、自动报告生成、实验管理和人机交互能力。

本方案的核心原则是：

```text
不改变现有 SLS 主模型的判别结果，不影响现有复现流程；
大模型作为“分析层、解释层、报告层、交互层”接入。
```

也就是说，大模型不直接替代 XLS-R + SLS 主模型，也不参与 EER 计算本身，而是围绕已有模型输出做增强。

## 2. 建议接入的大模型功能

结合当前项目情况，建议优先实现以下 5 类功能。

## 2.1 自动实验报告生成

### 功能描述

根据一次实验的配置、score 文件、EER 结果、数据集规模和 key 分布，自动生成 Markdown 格式实验报告。

输入包括：

- 运行脚本名称，例如 `scripts/eval_df_20000.sh`
- protocol 文件，例如 `ASVspoof2021.DF.cm.eval.first20000.trl.txt`
- score 文件，例如 `scores/scores_DF_20000.txt`
- key 文件，例如 `keys/DF_20000/CM/trial_metadata.txt`
- EER 结果，例如 `1.99%`
- score 分布统计，例如均值、标准差、最大值、最小值

输出示例：

```text
reports/df_20000_report.md
```

报告内容包括：

- 实验目的
- 数据规模
- phase 分布
- bonafide/spoof 标签分布
- score 文件校验
- EER 结果
- 与论文报告结果对比
- 可复现实验命令
- 注意事项

### 价值

当前项目已经有 `复现报告.md`，但仍然需要人工整理。引入大模型后，可以把每次实验自动整理成统一格式，减少人工写报告的时间。

## 2.2 推理结果解释与样本分析

### 功能描述

对某次推理生成的 score 文件进行统计分析，并让大模型生成自然语言解释。

例如对 `scores/scores_DF_20000.txt` 做：

- 分数分布统计
- 高置信 spoof 样本列表
- 高置信 bonafide 样本列表
- 接近阈值的疑难样本列表
- 按 label 统计 score 分布
- 错分样本分析，前提是有官方 key

大模型负责把统计结果解释成可读文本，例如：

```text
本次 20000 条子集测试中，eval phase 共 17436 条，其中 spoof 16933 条，bonafide 503 条。
模型整体 EER 为 1.99%，与仓库报告的 1.92% 接近。
从 score 分布看，大部分 spoof 样本得分集中在较低区间，说明模型对多数 spoof 样本具有较强区分能力。
```

### 价值

相比只给出一个 EER，解释报告可以帮助项目组理解模型行为，尤其适合后续答辩、论文复现报告和组会展示。

## 2.3 实验助手式问答

### 功能描述

提供一个命令行交互入口，让用户可以用自然语言询问当前项目状态。

示例问题：

```text
现在有哪些实验已经跑完？
20000 条测试的 EER 是多少？
5000 条和 20000 条结果差异大吗？
哪个 score 文件是中断产生的，不能用于报告？
如果我要重新跑 20000 条，该执行什么命令？
```

大模型读取项目中的结构化摘要文件后回答，而不是凭空生成。

建议维护一个机器可读的实验状态文件，例如：

```text
artifacts/experiment_registry.json
```

示例结构：

```json
{
  "df_20000": {
    "protocol": "database/ASVspoof_DF_cm_protocols/ASVspoof2021.DF.cm.eval.first20000.trl.txt",
    "score": "scores/scores_DF_20000.txt",
    "key": "keys/DF_20000/CM/trial_metadata.txt",
    "eer": 1.99,
    "status": "completed"
  }
}
```

### 价值

项目文件和脚本越来越多后，自然语言问答可以降低使用门槛。

## 2.4 错分样本辅助分析

### 功能描述

在拥有官方 key 和 score 的情况下，先用本地脚本找出错分或边界样本，再由大模型生成分析说明。

流程：

```text
score + key -> 计算阈值/错误样本 -> 生成结构化统计 -> 大模型生成解释
```

可输出：

- 错分 bonafide 样本
- 错分 spoof 样本
- 距离阈值最近的样本
- 高置信错分样本
- 每类错误的可能原因

需要注意：

大模型不能直接“听音频”判断真假，除非额外接入多模态音频模型。因此这里的大模型只解释统计结果和元数据，不替代反欺骗模型。

### 价值

该功能适合后续扩展实验分析，例如比较不同模型、不同数据子集、不同阈值策略。

## 2.5 自动生成运行命令与排错建议

### 功能描述

根据用户选择的数据规模、GPU 显存和目标任务，自动生成运行命令。

示例：

```text
我要跑 20000 条 DF 子集，用 4GB 显存，怎么跑？
```

系统返回：

```bash
source venv/bin/activate
bash scripts/eval_df_20000.sh
bash scripts/eer_df_20000.sh
```

如果日志中出现错误，例如：

```text
CUDNN_STATUS_NOT_INITIALIZED
CUDA out of memory
NoBackendError
```

大模型可以结合已知项目配置给出排错建议：

- 保持 `EVAL_BATCH_SIZE=1`
- 启用 `--disable_cudnn`
- 使用 `torchaudio` 读取 FLAC
- 确认 `MMpaper_model.pth` 是否存在
- 确认 `xlsr2_300m.pt` 是否存在

### 价值

减少重复沟通，把之前已经踩过的坑固化为项目内助手能力。

## 3. 推荐总体架构

建议采用“本地确定性计算 + 大模型文本增强”的架构。

```text
                  +----------------------+
                  | 用户/命令行/未来网页 |
                  +----------+-----------+
                             |
                             v
                  +----------------------+
                  | LLM Assistant Layer  |
                  | 大模型助手层         |
                  +----------+-----------+
                             |
              +--------------+--------------+
              |                             |
              v                             v
  +------------------------+    +---------------------------+
  | 本地分析工具层         |    | 项目知识与实验注册表      |
  | scripts / Python utils |    | reports / JSON / Markdown |
  +-----------+------------+    +-------------+-------------+
              |                               |
              v                               v
  +--------------------------------------------------------+
  | 现有 SLS 推理与评估流程                               |
  | main.py / model.py / evaluate_2021_DF.py / scores / keys |
  +--------------------------------------------------------+
```

核心思想：

- 所有数值计算、EER 计算、score 统计都由本地 Python 脚本完成。
- 大模型只读取结构化结果，并生成解释、报告或命令建议。
- 重要指标不由大模型“估计”，避免幻觉。

## 4. 建议新增目录结构

建议新增以下目录：

```text
llm/
  __init__.py
  client.py
  prompts.py
  report_generator.py
  experiment_assistant.py
  schemas.py

analysis/
  __init__.py
  score_stats.py
  error_analysis.py
  experiment_registry.py

artifacts/
  experiment_registry.json
  score_stats/

reports/
  df_5000_report.md
  df_20000_report.md
```

说明：

- `llm/client.py`：封装大模型 API 调用。
- `llm/prompts.py`：统一管理提示词模板。
- `llm/report_generator.py`：根据实验统计生成报告。
- `llm/experiment_assistant.py`：实现命令行问答助手。
- `analysis/score_stats.py`：本地计算 score 分布、标签分布、EER 结果。
- `analysis/error_analysis.py`：本地计算错分样本和边界样本。
- `analysis/experiment_registry.py`：维护实验登记表。
- `reports/`：保存大模型生成的中文报告。

## 5. 大模型服务接入方式

建议先把大模型服务做成可替换后端，不要和某一个厂商强绑定。

## 5.1 配置文件

新增：

```text
config/llm.yaml
```

示例：

```yaml
provider: openai_compatible
base_url: ${LLM_BASE_URL}
api_key_env: LLM_API_KEY
model: ${LLM_MODEL}
temperature: 0.2
max_tokens: 2000
```

环境变量：

```bash
export LLM_API_KEY="你的 API Key"
export LLM_BASE_URL="https://api.example.com/v1"
export LLM_MODEL="your-model-name"
```

这样可以支持：

- OpenAI-compatible API
- 本地部署的大模型服务
- 学校或实验室内网模型服务
- 后续替换成其他模型

## 5.2 API 封装

建议 `llm/client.py` 提供统一接口：

```python
class LLMClient:
    def generate(self, system_prompt: str, user_prompt: str) -> str:
        ...
```

项目内部只调用 `LLMClient.generate()`，不直接散落 API 调用代码。

### 价值

后续更换模型时，只需要改 `client.py`，不需要改报告生成和实验助手逻辑。

## 6. 数据流设计

## 6.1 生成实验统计

先由本地脚本读取 score 和 key：

```bash
venv/bin/python analysis/score_stats.py \
  --score scores/scores_DF_20000.txt \
  --key keys/DF_20000/CM/trial_metadata.txt \
  --phase eval \
  --output artifacts/score_stats/df_20000.json
```

输出 JSON 示例：

```json
{
  "experiment": "df_20000",
  "score_file": "scores/scores_DF_20000.txt",
  "num_scores": 20000,
  "phase": "eval",
  "num_eval_trials": 17436,
  "label_counts": {
    "spoof": 16933,
    "bonafide": 503
  },
  "eer": 1.99,
  "score_distribution": {
    "mean": -15.261319,
    "std": 3.285387,
    "min": -18.684095,
    "max": 0.0
  }
}
```

## 6.2 大模型生成报告

再调用：

```bash
venv/bin/python llm/report_generator.py \
  --stats artifacts/score_stats/df_20000.json \
  --output reports/df_20000_report.md
```

大模型只接收 JSON 统计结果，不直接遍历大型 score 文件。

好处：

- 降低 token 消耗。
- 降低幻觉风险。
- 报告中的数字来自本地脚本，可信度更高。

## 7. Prompt 设计原则

提示词要强调：

- 只能基于输入 JSON 中的数字写报告。
- 不允许编造未提供的实验结果。
- 不允许声称完整 benchmark 复现，除非输入中明确 `full_eval=true`。
- 对子集结果必须标明是 subset。
- 输出中文 Markdown。

示例 system prompt：

```text
你是一个音频深度伪造检测实验报告助手。
你只能基于用户提供的结构化实验数据写报告，不能编造任何指标。
如果实验数据是子集，请明确说明该结果不是完整 benchmark 结果。
输出应为中文 Markdown，语言严谨，适合论文复现报告或组会展示。
```

## 8. 命令行接口设计

建议新增统一入口：

```bash
venv/bin/python llm_assistant.py report --experiment df_20000
venv/bin/python llm_assistant.py ask "20000条实验结果怎么样？"
venv/bin/python llm_assistant.py troubleshoot logs/eval.log
```

子命令说明：

- `report`：生成实验报告。
- `ask`：项目状态问答。
- `troubleshoot`：根据日志给出排错建议。
- `compare`：比较两个实验，例如 `df_5000` 和 `df_20000`。

## 9. 与现有代码的集成点

## 9.1 不建议改动主推理路径

现有主推理路径是：

```text
main.py -> Model -> score file
```

不建议在这个路径中直接调用大模型，因为：

- 大模型调用慢，会拖慢推理。
- 网络/API 不稳定会影响主模型复现。
- 大模型输出不应该影响 EER 计算。

因此建议保持：

```text
推理阶段只生成 scores
分析阶段再调用大模型
```

## 9.2 建议接入位置

推荐接入点是推理完成之后：

```text
scores/scores_DF_20000.txt
keys/DF_20000/CM/trial_metadata.txt
        |
        v
analysis/score_stats.py
        |
        v
artifacts/score_stats/df_20000.json
        |
        v
llm/report_generator.py
        |
        v
reports/df_20000_report.md
```

## 10. 分阶段实施计划

## 第一阶段：本地统计层

目标：不接 API，先把结构化统计准备好。

任务：

- 新增 `analysis/score_stats.py`
- 支持读取 score + key
- 输出 JSON 统计
- 支持 5000 和 20000 实验
- 维护 `artifacts/experiment_registry.json`

验收标准：

- 能自动复算 `df_20000` 的 EER 为 `1.99`
- 能输出 score 分布、标签分布、phase 分布
- 不依赖网络

## 第二阶段：报告生成层

目标：接入大模型 API，生成中文报告。

任务：

- 新增 `llm/client.py`
- 新增 `llm/prompts.py`
- 新增 `llm/report_generator.py`
- 从 JSON 统计生成 `reports/df_20000_report.md`

验收标准：

- 报告中所有数字与 JSON 一致
- 明确说明 20000 条是子集结果
- 不编造完整 DF benchmark 结果

## 第三阶段：实验助手问答

目标：可以用自然语言查询项目状态。

任务：

- 新增 `llm/experiment_assistant.py`
- 读取 `experiment_registry.json`
- 支持问答：
  - 已完成哪些实验
  - EER 是多少
  - 如何重跑
  - 哪些结果不能用于报告

验收标准：

- 能正确回答 20000 条 EER 为 `1.99`
- 能提醒 `scores_DF_subset.txt` 是中断结果，不建议报告

## 第四阶段：错误样本分析

目标：提升实验分析深度。

任务：

- 新增 `analysis/error_analysis.py`
- 输出错分样本、边界样本、高置信样本
- 大模型基于统计结果生成解释

验收标准：

- 输出错误样本 CSV
- 输出中文错误分析 Markdown

## 11. 风险与注意事项

## 11.1 大模型幻觉风险

风险：

大模型可能编造不存在的实验结果或误读指标。

规避方式：

- 所有数值先由本地脚本计算。
- 大模型只读 JSON。
- prompt 中明确禁止编造。
- 报告生成后保留原始 JSON 作为依据。

## 11.2 隐私与数据安全

风险：

如果把完整 score、metadata 或音频路径上传给外部 API，可能涉及数据合规问题。

规避方式：

- 默认只上传汇总统计。
- 不上传原始音频。
- 不上传大规模逐条 score，除非明确需要。
- 对样本 ID 可做脱敏或只取少量示例。

## 11.3 API 稳定性

风险：

外部大模型 API 可能超时、限流、费用不可控。

规避方式：

- 加入重试机制。
- 设置超时。
- 缓存生成结果。
- 本地统计和 EER 不依赖大模型。

## 11.4 结果权威性

风险：

大模型生成的解释不应被视为模型性能指标。

规避方式：

- EER、score 统计、标签分布都以本地脚本输出为准。
- 大模型报告只作为辅助说明。

## 12. 推荐优先级

建议优先级如下：

```text
P0: analysis/score_stats.py，本地统计与 JSON 输出
P1: llm/report_generator.py，自动中文实验报告
P2: experiment_registry.json，实验状态管理
P3: 命令行问答助手
P4: 错分样本分析与解释
```

最小可行版本应包括：

- 一个统计脚本
- 一个大模型 client
- 一个报告生成脚本
- 一个示例报告

## 14. 总结

本项目的大模型接入不建议直接改造反欺骗主模型，而应作为实验分析和报告生成的扩展层。

推荐路线是：

```text
先本地统计 -> 再大模型生成报告 -> 再扩展问答和排错 -> 最后做错分样本解释
```

这种设计可以保证：

- 主模型复现结果稳定可信。
- 大模型能力可以逐步扩展。
- 后续更换大模型服务成本较低。
- 报告和问答结果有本地统计数据作为依据。

对于当前项目阶段，建议先实现 `analysis/score_stats.py` 和 `llm/report_generator.py`，用已经完成的 `scores/scores_DF_20000.txt` 和 `keys/DF_20000/CM/trial_metadata.txt` 生成第一版大模型辅助中文实验报告。
